🛠️ ¿Cómo funciona el sistema por dentro?

Una explicación visual y conceptual de la arquitectura CI/CD del proyecto: quién habla con quién, cómo fluye el código desde tu teclado hasta Kubernetes, y por qué cada pieza existe.

🐍 FastAPI 🐳 Docker 🤖 Jenkins ⎈️ Kubernetes 📦 Docker Hub 🐙 GitHub

1. La idea en una sola frase

🌟 Resumen de todo el sistema:

Cada vez que haces git push, Jenkins lo detecta automáticamente, corre los tests, construye una imagen Docker, la publica en Docker Hub, y Kubernetes descarga esa imagen nueva y reemplaza los pods sin que la app deje de responder. Tú solo haces push. El resto ocurre solo.
🏭 Analógia — La fábrica automática

Imagina que escribes una receta nueva (tu código) y la depositas en un buzón (GitHub). Un robot en la fábrica (Jenkins) revisa el buzón cada 5 minutos. Cuando encuentra una receta nueva: primero la prueba (tests), luego la empaqueta (imagen Docker), la envía al almacén central (Docker Hub), y por último le dice al equipo de reparto (Kubernetes) que empiece a entregar la nueva versión a los clientes (pods corriendo).

2. Los actores del sistema

El sistema tiene 6 actores. Tres viven en tu máquina (LOCAL) y dos viven en internet (NUBE). Cada uno tiene un rol único.

🐍
Tu app — LOCAL

FastAPI

La aplicación web que construiste. Expone una API REST con operaciones CRUD de tareas y un endpoint /health para que Kubernetes sepa si está viva.

Corre dentro de pods de Kubernetes en el puerto 8000.

🐳
Empaquetador — LOCAL

Docker

Convierte la aplicación en una imagen inmutable que incluye todo lo necesario para correr: Python, dependencias y el código. La imagen es el artefacto que viaja entre sistemas.

🤖
Robot CI — LOCAL

Jenkins

El servidor de automatización. Observa GitHub, ejecuta el pipeline con 5 stages y coordina toda la cadena. Corre en un contenedor Docker en tu máquina.

📦
Registry — ☁️ NUBE

Docker Hub

El almacén central de imágenes. Jenkins empuja la imagen construida aquí, y Kubernetes la descarga desde aquí para crear los pods. Es el puente entre los dos sistemas.

🐙
Fuente de verdad — ☁️ NUBE

GitHub

El repositorio de código. Contiene la aplicación, el Jenkinsfile, el Dockerfile y los manifiestos de Kubernetes. Cada push aquí dispara el pipeline.

⎈️
Orquestador — LOCAL

Kubernetes

El director de orquesta de contenedores. Mantiene 2 pods corriendo siempre, escala a 5 si hay mucha carga, reemplaza pods caídos y hace rolling updates sin downtime.

3. El flujo completo paso a paso

Desde que escribes código hasta que el usuario puede consumir la nueva versión de la API, ocurren 7 eventos encadenados. La mayoría son automáticos.

Explicación paso a paso

1
📁 Tú haces git push — MANUAL Terminas de escribir código (una nueva funcionalidad, un fix, un cambio de configuración) y ejecutas git add . && git commit -m "mi cambio" && git push origin main. El commit llega al repositorio en GitHub (internet).
⇓
2
🕵️ Jenkins detecta el cambio — AUTOMÁTICO Jenkins tiene un temporizador configurado con pollSCM('H/5 * * * *') que pregunta a GitHub cada 5 minutos: "¿hay commits nuevos?". En cuanto detecta uno, arranca el pipeline leyendo el Jenkinsfile del propio repositorio.
📩 Analogía
Como un cartero que revisa el buzón cada 5 minutos. Cuando encuentra carta, activa el proceso.
⇓
3
🧪 Stage Test — AUTOMÁTICO (dentro de Jenkins) Jenkins crea un entorno virtual Python, instala las dependencias del requirements.txt y ejecuta los 11 tests con pytest tests/ -v --junit-xml=test-report.xml. Si alguno falla, el pipeline se detiene y no hay build ni deploy. El reporte XML queda publicado en la UI de Jenkins como gráfico de tendencia.
⇓
4
🏗️ Stage Build — AUTOMÁTICO (Docker en tu máquina) Jenkins llama al daemon Docker de tu máquina (via el socket /var/run/docker.sock montado en el contenedor Jenkins) y ejecuta:
Stage Build
docker build -t jaquimbayoc/devops-taskapi:5 .
docker tag  jaquimbayoc/devops-taskapi:5 jaquimbayoc/devops-taskapi:latest
El número 5 es el BUILD_NUMBER de Jenkins, incrementa en cada ejecución. Siempre también se etiqueta como latest.
⇓
5
📤 Stage Push — AUTOMÁTICO (LOCAL → NUBE) Jenkins usa la credencial dockerhub-creds (almacenada de forma segura, nunca en logs) para autenticarse y subir la imagen a Docker Hub:
Stage Push
# La contraseña nunca aparece en los logs
echo "$DOCKER_PASS" | docker login -u "$DOCKER_USER" --password-stdin
docker push jaquimbayoc/devops-taskapi:5
docker push jaquimbayoc/devops-taskapi:latest
Ahora la imagen está en hub.docker.com disponible para cualquiera que tenga permisos.
⇓
6
🚀 Stage Deploy — AUTOMÁTICO (NUBE → LOCAL) Jenkins ejecuta comandos kubectl que le hablan al API server de Kubernetes (el que corre dentro de Docker Desktop). Hay un detalle técnico importante:
⚠️ El problema del kubeconfig:
Jenkins corre dentro de un contenedor Docker. El kubeconfig usa 127.0.0.1 como dirección del API server de Kubernetes. Pero desde dentro de un contenedor, 127.0.0.1 apunta al propio contenedor Jenkins, no al host.

La solución: El Jenkinsfile copia el kubeconfig y reemplaza 127.0.0.1 por host.docker.internal (nombre especial que Docker Desktop provee para llegar al host desde dentro de un contenedor).
Stage Deploy (fragmento)
# Copiar y parchar el kubeconfig
cp $HOME/.kube/config /tmp/kubeconfig
sed -i 's/127.0.0.1/host.docker.internal/g' /tmp/kubeconfig
export KUBECONFIG=/tmp/kubeconfig

# Aplicar manifiestos y actualizar imagen
kubectl apply -f k8s/ --validate=false
kubectl set image deployment/taskapi taskapi=jaquimbayoc/devops-taskapi:5 -n devops-project
kubectl rollout status deployment/taskapi -n devops-project --timeout=120s
Kubernetes descarga la imagen nueva de Docker Hub y hace un rolling update: levanta pods nuevos antes de terminar los viejos. La app nunca deja de responder.
⇓
7
👥 El usuario consume la nueva versión — TRANSPARENTE El Service de Kubernetes (tipo LoadBalancer) expone los pods en el puerto 80 del host. El usuario accede a http://localhost/docs o http://localhost/health y ya está usando la versión nueva, sin haber hecho nada manualmente.
⏱️ Tiempo total del ciclo: Desde el git push hasta que el nuevo código corre en Kubernetes: ~4 minutos (espera del poll ~3 min + pipeline ~1.5 min). Si se dispara manualmente con "Build Now", ~90 segundos.

4. Conectividad — ¿quién habla con quién y cómo?

Esta es la tabla completa de conexiones del sistema. Cada fila es una conexión real con su protocolo, dirección y razón de existir.

Origen → Destino Cómo / Puerto Para qué
💻 Tú (terminal) → ☁️ GitHub HTTPS :443 HTTPS git push sube el código nuevo. Dispara el ciclo completo.
💻 Jenkins → ☁️ GitHub HTTPS :443 HTTPS SCM polling (H/5 * * * *): Jenkins pregunta si hay commits nuevos cada 5 minutos. Si los hay, clona el repo.
💻 Jenkins (contenedor) → 💻 Docker daemon (host) Unix socket SOCKET Jenkins usa el Docker del host (no el del contenedor) para construir imágenes. Esto es el montaje /var/run/docker.sock:/var/run/docker.sock en jenkins/docker-compose.yml.
💻 Jenkins → ☁️ Docker Hub HTTPS :443 HTTPS docker push en el stage Push. Usa el PAT (Personal Access Token) almacenado como credencial en Jenkins con ID dockerhub-creds.
💻 Jenkins (contenedor) → 💻 Kubernetes API server HTTPS :6443 TCP kubectl apply en el stage Deploy. La conexión va a host.docker.internal:6443 (el API server de Kubernetes de Docker Desktop). El kubeconfig se parchea para cambiar 127.0.0.1 por host.docker.internal.
💻 Kubernetes (pods) → ☁️ Docker Hub HTTPS :443 HTTPS docker pull automático cuando Kubernetes crea un pod nuevo. La imagen se descarga de Docker Hub con imagePullPolicy: Always, garantizando que siempre se use la versión más reciente.
💻 Tú (navegador) → 💻 Service K8s (LoadBalancer) HTTP :80 HTTP El Service de Kubernetes expone los pods en localhost:80. El Service balancea el tráfico entre Pod 1 y Pod 2 de forma transparente.
💻 Service K8s → 💻 Pods FastAPI TCP :8000 TCP El Service redirige el tráfico del puerto 80 al puerto 8000 de cada pod. FastAPI (Uvicorn) escucha en 0.0.0.0:8000 dentro del contenedor.
💻 Kubernetes (kubelet) → 💻 Pod FastAPI /health HTTP :8000 REST Liveness probe y readiness probe. Kubernetes pregunta a cada pod GET /health periódicamente. Si responde {"status":"ok"} con HTTP 200, el pod está sano. Si no, Kubernetes reinicia el pod.
💻 Tú (navegador) → 💻 Jenkins UI HTTP :8080 HTTP La interfaz web de Jenkins donde ves el estado del pipeline, los logs de cada stage y el reporte JUnit.

El truco del Docker socket

Jenkins corre dentro de un contenedor Docker. Pero el stage Build necesita ejecutar docker build. ¿Cómo?

La solución es montar el socket Unix del daemon Docker del host dentro del contenedor Jenkins:

jenkins/docker-compose.yml
volumes:
  # Montar el socket del host dentro de Jenkins
  - /var/run/docker.sock:/var/run/docker.sock
  - ~/.kube:/var/jenkins_home/.kube:ro

Cuando Jenkins ejecuta docker build, los comandos llegan al daemon Docker del host, no a un daemon dentro del contenedor Jenkins (que no existe). Las imágenes se construyen en el host y están disponibles localmente.

Docker-in-Docker vs Socket

❌ Docker-in-Docker (DinD):
Correr un daemon Docker dentro del contenedor Jenkins. Complejo, inseguro y lento.

✅ Socket mounting (este proyecto):
El contenedor Jenkins comparte el daemon Docker del host via el socket Unix. Simple, rápido y estándar en entornos de desarrollo.

🏭 Analogía
Como si el robot (Jenkins) no tuviera sus propias herramientas, pero tiene acceso al taller del dueño de casa (el host) a través de una ventanilla (el socket).

El truco del kubeconfig

Problema de red

Kubeconfig original:

~/.kube/config (fragmento)
server: https://127.0.0.1:6443
# ↑ funciona desde el host, pero...
# desde el contenedor Jenkins, 127.0.0.1
# apunta al propio Jenkins, ¡no al host!

Solución en el Jenkinsfile:

Stage Deploy
cp $HOME/.kube/config /tmp/kubeconfig
# Reemplazar 127.0.0.1 por la dirección del host
sed -i 's/127.0.0.1/host.docker.internal/g' /tmp/kubeconfig
export KUBECONFIG=/tmp/kubeconfig

host.docker.internal es un nombre DNS especial que Docker Desktop inyecta en cada contenedor. Resuelve a la dirección IP del host (tu máquina), permitiendo al contenedor Jenkins comunicarse con el API server de Kubernetes que corre en el host.

🏭 Analogía
Estás en un hotel (el contenedor Jenkins) y quieres llamar a la recepción del edificio (Kubernetes). Si marcas el 127.0.0.1 llamas al teléfono de tu propia habitación. host.docker.internal es como marcar el número directo de recepción del edificio.

Este parche se hace en una copia temporal (/tmp/kubeconfig) para no modificar el kubeconfig original que Jenkins tiene montado en modo de solo lectura.

5. Conceptos clave con ejemplos simples

🏗️ Imagen Docker vs Contenedor

📚 Analogía
La imagen es el molde de galletas. El contenedor es la galleta resultante. Con el mismo molde puedes hacer miles de galletas idénticas. Romper una galleta no afecta las demás ni el molde.

La imagen es inmutable (no cambia) y está guardada en Docker Hub. El contenedor es una instancia corriendo de esa imagen. Kubernetes crea contenedores (pods) a partir de la imagen.

Cada vez que Jenkins termina el stage Build, produce una nueva imagen etiquetada con el número de build (:5, :6...) y también como :latest.

Imagen vs Contenedor
Relación imagen ↔ contenedores
# Una imagen → múltiples contenedores
devops-taskapi:5  # imagen (en Docker Hub)
    ├─ Pod 1  # contenedor corriendo
    ├─ Pod 2  # contenedor corriendo
    └─ Pod 3  # si HPA escala

# Verificar imágenes locales
docker images
REPOSITORY                 TAG   IMAGE ID
devops-taskapi             5     a1b2c3d4e5f6
devops-taskapi             latest a1b2c3d4e5f6

📦 ¿Por qué necesitamos Docker Hub?

Problema: Jenkins construye la imagen en tu máquina. Kubernetes también corre en tu máquina. ¿No podrían comunicarse directamente?

En este proyecto sí, técnicamente podrían. Pero usamos Docker Hub para practicar el flujo real de producción, donde Jenkins y Kubernetes están en máquinas diferentes (un servidor CI y un clúster en la nube). Docker Hub es el punto neutral donde ambos se encuentran: Jenkins empuja, Kubernetes jala.

⎈️ Kubernetes — sus piezas en este proyecto

📁 Namespace

Un espacio de nombres aislado dentro del clúster. Todos los recursos del proyecto viven en devops-project.

Es como una carpeta: organiza y aísla los recursos de un proyecto del resto del clúster.

🚀 Deployment

Declara cuántos pods deben correr y cón qué imagen. Kubernetes garantiza que siempre haya exactamente 2 pods corriendo.

Si un pod cae, el Deployment levanta uno nuevo automáticamente.

🔌 Service

Expone los pods con una dirección estable. Sin él, los pods tienen IPs que cambian cada vez que se recrean.

Tipo LoadBalancer: mapea localhost:80 → pods en puerto 8000 y balancea el tráfico.

📜 ConfigMap

Variables de entorno inyectadas en los pods. Permite cambiar configuración sin reconstruir la imagen.

En el proyecto: PYTHONUNBUFFERED=1 y APP_ENV=production.

📈 HPA

Horizontal Pod Autoscaler. Monitoriza el uso de CPU de los pods. Si supera el 60%, escala de 2 a máximo 5 pods.

Cuando la carga baja, reduce los pods automáticamente.

🏥 Probes

Liveness: ¿el pod está vivo? Si falla, lo reinicia.
Readiness: ¿el pod está listo para recibir tráfico? Si falla, lo quita del Service.

Ambas consultan GET /health → {"status":"ok"}.

🤖 Pipeline declarativo de Jenkins

El Jenkinsfile define el pipeline como código versionado en el repositorio. Esto se llama Pipeline as Code.

Jenkinsfile — estructura resumida
pipeline {
    agent any                          // correr en cualquier agente disponible
    environment {
        IMAGE_NAME = "usuario/devops-taskapi"
    }
    triggers { pollSCM('H/5 * * * *') }  // revisar GitHub cada 5 min

    stages {
        stage('Checkout') { … }           // ① clonar código
        stage('Test')     { … }           // ② pytest 11 tests
        stage('Build')    { … }           // ③ docker build
        stage('Push')     { … }           // ④ docker push → Docker Hub
        stage('Deploy')   { … }           // ⑤ kubectl apply → K8s
    }

    post {
        always  { // siempre: limpiar workspace y cerrar sesión Docker }
        success { // solo si todo verde: log de éxito }
        failure { // solo si hay error: log de fallo }
    }
}
🌟 Por qué el orden de los stages importa:
Si los tests fallan en el stage Test, el pipeline se detiene. Nunca se construye ni se sube una imagen de código roto. Esto garantiza que solo código probado llega a producción.

🐉 El Dockerfile multistage

El Dockerfile usa dos etapas para producir una imagen mínima y segura:

  • builder: instala todas las dependencias en un directorio temporal.
  • runtime: copia solo los archivos instalados (no las herramientas de compilación) a una imagen Python slim limpia. La imagen final es más pequeña y no expone herramientas de construcción.

Además, la app corre con un usuario sin privilegios (appuser). Si alguien explota una vulnerabilidad en la app, no tiene acceso root al contenedor.

Dockerfile multistage
Dockerfile
# Etapa 1: instalar deps
FROM python:3.12-slim AS builder
COPY requirements.txt .
RUN pip install --prefix=/install -r requirements.txt

# Etapa 2: imagen mínima
FROM python:3.12-slim AS runtime
COPY --from=builder /install /usr/local
COPY app/ ./app/
RUN adduser --disabled-password appuser
USER appuser          # sin privilegios
CMD ["uvicorn", "app.main:app", ...]

6. Preguntas frecuentes

¿Por qué Jenkins no puede simplemente hacer kubectl apply sin el parche del kubeconfig? +
Porque Jenkins corre dentro de un contenedor Docker. El archivo kubeconfig le dice a kubectl que el API server de Kubernetes está en 127.0.0.1:6443. Pero desde dentro de un contenedor, 127.0.0.1 (localhost) apunta al propio contenedor, no al host donde Kubernetes realmente corre. El parche con host.docker.internal corrige la dirección para que apunte al host correctamente.
¿Qué pasa si un test falla? +
El pipeline se detiene en el stage Test. Los stages Build, Push y Deploy no se ejecutan. La versión anterior sigue corriendo en Kubernetes sin cambios. Jenkins marca el build como FAILED y publica el reporte JUnit mostrando qué test falló. La app en producción nunca ve código roto.
¿Qué es el emptyDir en el Deployment y por qué se usa? +
emptyDir es un volumen temporal que vive mientras el pod existe. Se usa para que la base de datos SQLite (/app/data/tasks.db) no se pierda si el contenedor se reinicia por un crash (solo si el pod se elimina se pierde). En producción real se usaría un PersistentVolumeClaim con almacenamiento externo. Para este entorno de aprendizaje, emptyDir es suficiente porque los datos son de prueba.
¿Por qué imagePullPolicy: Always en el Deployment? +
Porque el tag latest no cambia entre builds, solo cambia a qué imagen apunta. Sin imagePullPolicy: Always, si Kubernetes ya tiene la imagen latest cacheada localmente, la usará aunque Docker Hub tenga una versión más nueva. Con Always, Kubernetes siempre verifica con Docker Hub y descarga la última versión antes de crear cada pod. Esto garantiza que cada deploy use el código recién pusheado.
¿Cómo sabe Kubernetes que un pod está sano? +
A través de las probes definidas en el Deployment:

Liveness probe: Kubernetes hace GET /health al pod cada 10 segundos. Si falla 3 veces consecutivas, reinicia el pod. Detecta estados bloqueados (deadlocks).

Readiness probe: Misma consulta, pero determina si el pod está listo para recibir tráfico del Service. Mientras el pod no esté listo, el Service no le envía peticiones. Esto evita que los usuarios reciban errores durante el arranque de un pod nuevo.
¿Qué significa que el HPA escala al 60% de CPU? +
El HPA (Horizontal Pod Autoscaler) monitoriza el uso de CPU de todos los pods del Deployment. Si el promedio supera el 60% del límite de CPU definido en el Deployment, añade un pod más (hasta el máximo de 5). Si la carga baja, reduce los pods (hasta el mínimo de 2).

Por ejemplo: si cada pod tiene límite de 200m CPU y el promedio de los 2 pods es 130m (65%), el HPA escala a 3 pods para que el promedio baje a ~87m (43%), por debajo del umbral.
¿Qué es un rolling update y por qué no hay downtime? +
Un rolling update es la estrategia de actualización por defecto de Kubernetes. En lugar de bajar todos los pods y levantar los nuevos (lo que causaría downtime), Kubernetes levanta un pod nuevo con la imagen actualizada, espera a que esté Ready (readiness probe), y solo entonces elimina un pod viejo. Repite hasta actualizar todos.

Con 2 réplicas: siempre hay al menos 1 pod corriendo durante el deploy. Los usuarios no notan la transición.
