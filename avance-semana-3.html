Namespace devops-project

Aislamiento lógico de todos los recursos Kubernetes del proyecto

Un Namespace en Kubernetes es una partición lógica del cluster. Agrupa todos los recursos del proyecto (Deployment, Service, HPA, ConfigMap) y permite aplicar cuotas, políticas de red y RBAC de forma independiente.

k8s/namespace.yaml 
apiVersion: v1
kind: Namespace
metadata:
  name: devops-project
  labels:
    project: devops-taskapi
terminal — aplicar y verificar 
kubectl apply -f k8s/namespace.yaml
namespace/devops-project created

kubectl get namespaces
NAME              STATUS   AGE
devops-project    Active   5s
default           Active   2d
💡 Todos los recursos usan namespace: devops-project
El campo metadata.namespace debe estar presente en cada manifiesto para que kubectl apply -f k8s/ los cree en el namespace correcto. Los comandos kubectl get también requieren el flag -n devops-project.

Deployment — 2 réplicas + liveness & readiness probes

Alta disponibilidad con health checks automáticos

El Deployment define la especificación de los pods: imagen Docker, número de réplicas, recursos (CPU/memoria), y las probes de salud que Kubernetes usa para gestionar el ciclo de vida de los contenedores.

k8s/deployment.yaml 
apiVersion: apps/v1
kind: Deployment
metadata:
  name: taskapi
  namespace: devops-project
spec:
  replicas: 2
  selector:
    matchLabels:
      app: taskapi
  template:
    metadata:
      labels:
        app: taskapi
    spec:
      containers:
        - name: taskapi
          image: TU-USUARIO/devops-taskapi:latest
          imagePullPolicy: Always
          ports:
            - containerPort: 8000
          envFrom:
            - configMapRef:
                name: taskapi-config
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "250m"
              memory: "256Mi"
          livenessProbe:
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 15
            periodSeconds: 20
            failureThreshold: 3
          readinessProbe:
            httpGet:
              path: /health
              port: 8000
            initialDelaySeconds: 5
            periodSeconds: 10
            failureThreshold: 3
          volumeMounts:
            - name: sqlite-data
              mountPath: /app/data
      volumes:
        - name: sqlite-data
          emptyDir: {}
💡 Liveness vs Readiness probe
  • livenessProbe: si falla 3 veces seguidas, Kubernetes reinicia el contenedor. Detecta procesos bloqueados.
  • readinessProbe: si falla, Kubernetes retira el pod del balanceo (no envía tráfico) pero no lo reinicia. El endpoint GET /health implementado en la Semana 1 sirve para ambas.
💡 requests vs limits
requests es lo que Kubernetes garantiza al pod en el momento del scheduling. limits es el máximo que puede consumir; si lo supera, el contenedor es throttled (CPU) o terminado (memoria). El HPA del paso 20 usa el valor de requests.cpu como base para calcular el porcentaje de utilización.
💡 emptyDir para SQLite en Kubernetes
La aplicación escribe la base de datos SQLite en /app/data. El volumen emptyDir crea un directorio temporal vinculado al ciclo de vida del pod: persiste mientras el pod vive (útil en desarrollo/aprendizaje) pero se borra si el pod se elimina o se mueve de nodo. En producción real se usaría un PersistentVolumeClaim o una base de datos externa (PostgreSQL, RDS, etc.).

Service — LoadBalancer puerto 80 → 8000

Acceso externo estable independiente de qué pods están activos

El Service expone el Deployment al exterior. Con Docker Desktop, el tipo LoadBalancer asigna automáticamente la IP localhost, por lo que la API queda accesible en http://localhost/docs sin configuración adicional.

k8s/service.yaml 
apiVersion: v1
kind: Service
metadata:
  name: taskapi-service
  namespace: devops-project
  labels:
    app: taskapi
spec:
  selector:
    app: taskapi      # enruta a los pods con este label
  ports:
    - protocol: TCP
      port: 80          # puerto externo
      targetPort: 8000  # puerto del contenedor
  type: LoadBalancer
💡 LoadBalancer en Docker Desktop
Docker Desktop provisiona automáticamente la IP localhost para los Services de tipo LoadBalancer. En un cluster cloud (EKS, GKE, AKS) se asignaría una IP pública. En Minikube se usaría minikube tunnel.

ConfigMap — Variables de entorno de la aplicación

Configuración desacoplada del código y de la imagen Docker

El ConfigMap almacena pares clave-valor que se inyectan como variables de entorno en los contenedores. Separa la configuración del código, lo que permite cambiar parámetros sin reconstruir la imagen.

k8s/configmap.yaml 
apiVersion: v1
kind: ConfigMap
metadata:
  name: taskapi-config
  namespace: devops-project
data:
  PYTHONUNBUFFERED: "1"
  APP_ENV: "production"
⚠️ ConfigMap no es para secretos
Las variables en un ConfigMap son texto plano visible para cualquiera con acceso al namespace. Para credenciales (contraseñas, tokens de API, claves) usa siempre un Secret de Kubernetes, que almacena los valores codificados en base64 y permite control de acceso más estricto.

HPA — Horizontal Pod Autoscaler (2–5 pods al 60% CPU)

Escalado automático en respuesta a la carga real del sistema

El HPA monitorea el consumo de CPU de los pods y ajusta automáticamente el número de réplicas del Deployment. Mantiene al menos 2 pods activos y puede escalar hasta 5 cuando la CPU supera el 60 % del valor de requests.cpu.

k8s/hpa.yaml 
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: taskapi-hpa
  namespace: devops-project
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: taskapi
  minReplicas: 2
  maxReplicas: 5
  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 60
💡 autoscaling/v2 y métricas de CPU
El HPA requiere que el cluster tenga el servidor de métricas (metrics-server) instalado. En Docker Desktop ya viene incluido. El cálculo es:

réplicas deseadas = ceil(uso_actual_cpu / (requests.cpu × 0.60))

Con requests.cpu = 100m y umbral del 60 %, el HPA escala hacia arriba cuando el uso supera los 60m (milli-CPUs) por pod en promedio.

Stage Deploy — kubectl apply + rollout sin downtime

Jenkins actualiza Kubernetes automáticamente tras cada build exitoso

El stage Deploy se añade al Jenkinsfile existente. Aplica todos los manifiestos de k8s/, actualiza la imagen del Deployment con el número de build exacto y espera a que el rollout termine antes de dar el stage por completado.

Cambios en jenkins/Dockerfile — añadir kubectl

jenkins/Dockerfile — bloque kubectl (nuevo) 
# ── kubectl ─────────────────────────────────────────────────
RUN curl -fsSL "https://dl.k8s.io/release/$(curl -fsSL https://dl.k8s.io/release/stable.txt)/bin/linux/amd64/kubectl" \
        -o /usr/local/bin/kubectl && \
    chmod +x /usr/local/bin/kubectl

Cambios en jenkins/docker-compose.yml — montar kubeconfig

jenkins/docker-compose.yml 
services:
  jenkins:
    build: .
    volumes:
      - jenkins_home:/var/jenkins_home
      - /var/run/docker.sock:/var/run/docker.sock
      - ~/.kube:/var/jenkins_home/.kube:ro   # kubeconfig de Docker Desktop

volumes:
  jenkins_home:

Stage Deploy en el Jenkinsfile

Jenkinsfile — stage Deploy 
        stage('Deploy') {
            steps {
                sh """
                    # 1. Copiar kubeconfig y adaptar para Docker Desktop
                    cp \$HOME/.kube/config /tmp/kubeconfig
                    sed -i 's/127.0.0.1/host.docker.internal/g' /tmp/kubeconfig
                    sed -i 's/certificate-authority-data:.*/insecure-skip-tls-verify: true/' /tmp/kubeconfig
                    export KUBECONFIG=/tmp/kubeconfig

                    # 2. Aplicar todos los manifiestos
                    kubectl apply -f k8s/ --validate=false

                    # 3. Actualizar imagen con el número de build exacto
                    kubectl set image deployment/taskapi taskapi=${IMAGE_NAME}:${BUILD_NUMBER} -n devops-project

                    # 4. Esperar a que el rollout termine (máx. 2 min)
                    kubectl rollout status deployment/taskapi -n devops-project --timeout=120s
                """
            }
        }
⚠️ Por qué copiar y modificar el kubeconfig
Desde dentro del contenedor Jenkins, 127.0.0.1 apunta al propio contenedor, no al host donde corre Docker Desktop. El sed reemplaza esa dirección por host.docker.internal (alias de red que Docker resuelve al host). El segundo sed elimina la verificación del certificado TLS, que no es válida para ese hostname alternativo. Sin estos dos pasos, kubectl devuelve “connection refused” o “x509 certificate invalid”.
💡 kubectl set image vs editar deployment.yaml
El manifiesto deployment.yaml tiene image: TU-USUARIO/devops-taskapi:latest como valor base. El comando kubectl set image sobrescribe ese valor en el cluster con la etiqueta versionada (:${BUILD_NUMBER}), garantizando que cada deploy usa la imagen exacta producida en ese build. El flag imagePullPolicy: Always del Deployment asegura que Kubernetes descarga la imagen nueva en cada reinicio de pod.
⚠️ Reconstruir la imagen Jenkins
Después de modificar jenkins/Dockerfile y jenkins/docker-compose.yml, es necesario reconstruir el contenedor Jenkins: 
cd jenkins
docker compose down
docker compose up -d --build

🎉 Proyecto Integrador completado
El pipeline CI/CD de 5 stages está operativo de extremo a extremo: cada git push ejecuta los tests, construye la imagen, la publica en Docker Hub y la despliega en Kubernetes sin downtime. El HPA escala la aplicación automáticamente entre 2 y 5 pods según la demanda real de CPU.

📄 Jenkinsfile completo (5 stages)

Jenkinsfile 
pipeline {
    agent any

    environment {
        IMAGE_NAME = "TU-USUARIO/devops-taskapi"  // ← reemplaza TU-USUARIO por tu Docker ID
    }

    triggers {
        pollSCM('H/5 * * * *')
    }

    stages {

        stage('Checkout') {
            steps { checkout scm }
        }

        stage('Test') {
            steps {
                sh '''
                    python3 -m venv .venv
                    . .venv/bin/activate
                    pip install --quiet -r requirements.txt
                    pytest tests/ -v --junit-xml=test-report.xml
                '''
            }
            post {
                always { junit 'test-report.xml' }
            }
        }

        stage('Build') {
            steps {
                sh "docker build -t ${IMAGE_NAME}:${BUILD_NUMBER} ."
                sh "docker tag  ${IMAGE_NAME}:${BUILD_NUMBER} ${IMAGE_NAME}:latest"
            }
        }

        stage('Push') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'dockerhub-creds',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )]) {
                    sh '''
                        echo "$DOCKER_PASS" | docker login -u "$DOCKER_USER" --password-stdin
                        docker push ${IMAGE_NAME}:${BUILD_NUMBER}
                        docker push ${IMAGE_NAME}:latest
                    '''
                }
            }
        }

        stage('Deploy') {
            steps {
                sh """
                    cp \$HOME/.kube/config /tmp/kubeconfig
                    sed -i 's/127.0.0.1/host.docker.internal/g' /tmp/kubeconfig
                    sed -i 's/certificate-authority-data:.*/insecure-skip-tls-verify: true/' /tmp/kubeconfig
                    export KUBECONFIG=/tmp/kubeconfig
                    kubectl apply -f k8s/ --validate=false
                    kubectl set image deployment/taskapi taskapi=${IMAGE_NAME}:${BUILD_NUMBER} -n devops-project
                    kubectl rollout status deployment/taskapi -n devops-project --timeout=120s
                """
            }
        }
    }

    post {
        always {
            sh 'docker logout || true'
            cleanWs()
        }
        success {
            echo "Pipeline OK — imagen ${IMAGE_NAME}:${BUILD_NUMBER} publicada y desplegada en Kubernetes"
        }
        failure {
            echo "Pipeline FAILED en build #${BUILD_NUMBER} — revisar logs"
        }
    }
}

Credenciales Docker Hub en Jenkins  💻 LOCAL + ☁️ NUBE

Solo en la primera vez — el volumen las persiste entre sesiones

El stage Push del Jenkinsfile necesita el credential ID dockerhub-creds. Si ya lo creaste en una sesión anterior, sáltate este paso.

☁️ NUBE — hub.docker.com (obtener el PAT)
Si aún no tienes un Personal Access Token de Docker Hub, créalo ahora:
  1. Inicia sesión en hub.docker.com.
  2. Haz clic en tu avatar → Account Settings → Security → Personal access tokens.
  3. Clic en Generate new token:
    • Description: jenkins-pipeline
    • Permissions: Read & Write
    • Clic en Generate y copia el token — solo se muestra una vez.
  1. En Jenkins: Manage Jenkins → Credentials → (global) → Add Credentials.
  2. Tipo: Username with password.
  3. Username: tu Docker ID (p. ej. jaquimbayoc).
  4. Password: el Personal Access Token (PAT) generado en Docker Hub con permisos Read & Write.
  5. ID: dockerhub-creds — debe ser exactamente este valor (el Jenkinsfile lo referencia así).
  6. Haz clic en Create.
💡 Por qué PAT y no contraseña de Docker Hub
Desde 2022, Docker Hub bloquea el login con contraseña directa desde CI. El Personal Access Token (PAT) es la forma correcta y permite revocar acceso al pipeline sin cambiar tu contraseña principal.
