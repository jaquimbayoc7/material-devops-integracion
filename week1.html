Capa de base de datos con SQLAlchemy

app/database.py — ORM, motor SQLite, sesión, modelo TaskDB

El archivo app/database.py centraliza toda la configuración de persistencia. Usa SQLite como motor (no necesita servidor separado) y SQLAlchemy como ORM para mapear objetos Python a filas de la tabla tasks.

app/database.py
"""Database engine, session factory, ORM model, and get_db dependency."""
from collections.abc import Generator

from sqlalchemy import Boolean, Column, Integer, String, create_engine
from sqlalchemy.orm import Session, declarative_base, sessionmaker

DATABASE_URL = "sqlite:///./tasks.db"

engine = create_engine(DATABASE_URL, connect_args={"check_same_thread": False})
SessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
Base = declarative_base()


class TaskDB(Base):
    __tablename__ = "tasks"

    id          = Column(Integer, primary_key=True, index=True)
    title       = Column(String(200), nullable=False)
    description = Column(String(500), nullable=True)
    completed   = Column(Boolean, default=False)


def get_db() -> Generator[Session, None, None]:
    db = SessionLocal()
    try:
        yield db
    finally:
        db.close()
💡 Por qué check_same_thread: False:
SQLite por defecto solo permite acceso desde el hilo que creó la conexión. FastAPI usa threading, por lo que esta opción es necesaria para evitar errores en producción. En MySQL/PostgreSQL no aplica, pero es correcto para SQLite en desarrollo.
💡 PEP 8 — orden de imports y anotación de tipo:
Los imports siguen el orden: stdlib → terceros → locales (PEP 8 §Imports). Generator[Session, None, None] usa el tipo de collections.abc (PEP 585) en lugar de typing.Generator, disponible desde Python 3.9. La anotación hace explícita la intención de la función para herramientas como mypy.
💡 Patrón yield en get_db():
La función es un generador. FastAPI la inyecta como dependencia en cada endpoint, abre la sesión, la cede al endpoint, y la cierra automáticamente en el bloque finally independientemente de si hubo excepción o no.

Modelos Pydantic — app/models.py

Los modelos definen y validan la estructura de datos que entran y salen de la API. Pydantic v2 valida automáticamente tipos, convierte datos y genera el esquema JSON para Swagger.

app/models.py
"""Pydantic schemas for request validation and response serialization."""
from pydantic import BaseModel, ConfigDict


class TaskCreate(BaseModel):
    title: str
    description: str | None = None


class TaskUpdate(BaseModel):
    title:       str  | None = None
    description: str  | None = None
    completed:   bool | None = None


class TaskResponse(BaseModel):
    id:          int
    title:       str
    description: str | None = None
    completed:   bool

    model_config = ConfigDict(from_attributes=True)  # Pydantic v2: leer desde ORM objects
💡 PEP 604 — X | None en lugar de Optional[X]:
Desde Python 3.10 (PEP 604), la sintaxis str | None es la forma preferida sobre Optional[str] de typing. Es más legible y no requiere importar Optional. Pydantic v2 soporta ambas formas, pero la nueva es la recomendada.

💡 ConfigDict vs class Config:
En Pydantic v2, la forma recomendada es model_config = ConfigDict(...). class Config sigue funcionando pero está deprecada y se eliminará en Pydantic v3.

Endpoints CRUD — app/routers/tasks.py

El router agrupa los 5 endpoints bajo el prefijo /tasks. Cada uno recibe la sesión de BD por inyección de dependencias (Depends(get_db)).

POST/tasks/Crear nueva tarea201 Created
GET/tasks/{id}Obtener tarea por ID200 / 404
PUT/tasks/{id}Actualizar tarea (parcial)200 / 404
DELETE/tasks/{id}Eliminar tarea204 / 404
app/routers/tasks.py (fragmento)
"""CRUD endpoints for the /tasks resource."""
from fastapi import APIRouter, Depends, HTTPException
from sqlalchemy.orm import Session

from app.database import TaskDB, get_db
from app.models import TaskCreate, TaskResponse, TaskUpdate

router = APIRouter(prefix="/tasks", tags=["Tasks"])


@router.get("/", response_model=list[TaskResponse])  # PEP 585: list[...] sin import
def list_tasks(db: Session = Depends(get_db)):
    return db.query(TaskDB).all()


@router.post("/", response_model=TaskResponse, status_code=201)
def create_task(task: TaskCreate, db: Session = Depends(get_db)):
    db_task = TaskDB(title=task.title, description=task.description)
    db.add(db_task)
    db.commit()
    db.refresh(db_task)
    return db_task


@router.put("/{task_id}", response_model=TaskResponse)
def update_task(task_id: int, updates: TaskUpdate, db: Session = Depends(get_db)):
    task = db.query(TaskDB).filter(TaskDB.id == task_id).first()
    if task is None:
        raise HTTPException(status_code=404, detail="Tarea no encontrada")
    for field, value in updates.model_dump(exclude_unset=True).items():
        setattr(task, field, value)  # aplica solo los campos enviados
    db.commit()
    db.refresh(task)
    return task


@router.delete("/{task_id}", status_code=204)
def delete_task(task_id: int, db: Session = Depends(get_db)):
    task = db.query(TaskDB).filter(TaskDB.id == task_id).first()
    if task is None:
        raise HTTPException(status_code=404, detail="Tarea no encontrada")
    db.delete(task)
    db.commit()
💡 PEP 585 — list[T] en lugar de List[T]:
Desde Python 3.9 (PEP 585), los tipos genéricos built-in como list, dict y tuple pueden usarse directamente como tipos genéricos. Ya no es necesario importar List de typing. FastAPI y Pydantic v2 soportan esta sintaxis.

💡 model_dump(exclude_unset=True) + setattr:
En lugar de tres if field is not None separados, iteramos solo sobre los campos que el cliente realmente envió en el body (excluye los omitidos). Esto hace el código escalable: si se agrega un campo nuevo al modelo, el endpoint lo aplica automáticamente sin modificar la lógica.

Entrypoint — app/main.py

app/main.py
"""FastAPI application entry point."""
from fastapi import FastAPI

from app.database import Base, engine
from app.routers import tasks

Base.metadata.create_all(bind=engine)

app = FastAPI(
    title="DevOps Task Manager API",
    description="API REST de gestión de tareas — Proyecto Integrador DevOps",
    version="1.0.0",
)

app.include_router(tasks.router)


@app.get("/health", tags=["Health"])
def health_check():
    """Endpoint de salud usado por liveness y readiness probes de Kubernetes."""
    return {"status": "ok"}
💡 Por qué el endpoint /health es clave para DevOps:
Kubernetes usa este endpoint para las liveness probes (el pod está vivo) y readiness probes (el pod está listo para recibir tráfico). Si el endpoint devuelve un estado no-2xx, Kubernetes reinicia el pod automáticamente.

Suite de tests con pytest

tests/test_tasks.py — 11 tests · BD aislada · inyección de dependencias

La suite de tests usa dependency injection override: reemplaza la dependencia get_db de la app con una que apunta a una base de datos en memoria. Así los tests nunca tocan tasks.db de producción y no dejan archivos residuales.

tests/test_tasks.py (estructura)
"""Integration tests for the Task Manager API."""
import pytest
from fastapi.testclient import TestClient
from sqlalchemy import create_engine
from sqlalchemy.orm import sessionmaker
from sqlalchemy.pool import StaticPool

from app.database import Base, get_db
from app.main import app

# BD en memoria: sin archivos residuales, conexiones compartidas via StaticPool
TEST_DATABASE_URL = "sqlite:///:memory:"

engine = create_engine(
    TEST_DATABASE_URL,
    connect_args={"check_same_thread": False},
    poolclass=StaticPool,
)
TestingSessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)


def override_get_db():
    db = TestingSessionLocal()
    try:
        yield db
    finally:
        db.close()


app.dependency_overrides[get_db] = override_get_db


@pytest.fixture(autouse=True)  # se ejecuta en cada test automáticamente
def setup_db():
    Base.metadata.create_all(bind=engine)
    yield
    Base.metadata.drop_all(bind=engine)  # limpia tras cada test


client = TestClient(app)


# Ejemplo de test
def test_health_check():
    r = client.get("/health")
    assert r.status_code == 200
    assert r.json() == {"status": "ok"}


def test_create_task():
    r = client.post("/tasks/", json={"title": "Mi tarea", "description": "Desc"})
    assert r.status_code == 201
    assert r.json()["title"] == "Mi tarea"
    assert r.json()["completed"] is False

💡 Por qué autouse=True en el fixture:
Asegura que antes de cada test la tabla se cree limpia y después se destruya. Esto hace que cada test sea completamente independiente — un principio clave del testing: los tests no deben depender del orden de ejecución.
💡 sqlite:///:memory: + StaticPool:
Usar una BD en memoria (:memory:) en lugar de un archivo (test_tasks.db) es la mejor práctica en testing: no deja archivos residuales, es más rápida y garantiza aislamiento total entre ejecuciones de la suite.
StaticPool asegura que todas las conexiones del motor compartan la misma base de datos en memoria — sin él, cada connect() crearía una BD nueva e independiente.

Dockerfile multistage

Imagen de producción slim · etapa builder · usuario sin privilegios

El build multistage es una práctica DevOps fundamental: separa la fase de instalación de dependencias (etapa builder) de la imagen final de runtime. El resultado es una imagen significativamente más pequeña y sin herramientas de compilación innecesarias.

Etapa builder
python:3.12-slim
pip install
→
COPY --from=builder
solo /install
→
Etapa runtime
python:3.12-slim
appuser (no root)
Dockerfile
# ── ETAPA 1: builder ─────────────────────────────────────
FROM python:3.12-slim AS builder

WORKDIR /build
COPY requirements.txt .

# Actualizar pip e instalar en /install (aislado del sistema)
RUN pip install --no-cache-dir --upgrade pip \
    && pip install --no-cache-dir --prefix=/install -r requirements.txt

# ── ETAPA 2: runtime ─────────────────────────────────────
FROM python:3.12-slim AS runtime

WORKDIR /app

# Copiar solo las dependencias instaladas
COPY --from=builder /install /usr/local

# Copiar código fuente
COPY app/ ./app/

# Puerto de la aplicación
EXPOSE 8000

# Usuario sin privilegios (seguridad)
RUN adduser --disabled-password --gecos "" appuser
USER appuser

CMD ["uvicorn", "app.main:app", "--host", "0.0.0.0", "--port", "8000"]
💡 Por qué usuario appuser (sin root):
OWASP recomienda ejecutar procesos en contenedores con el mínimo privilegio posible. Si un atacante explota la aplicación, no tiene acceso root al contenedor ni al host. Kubernetes también puede forzar esta política con securityContext.runAsNonRoot: true.

docker-compose para desarrollo local

Hot-reload · volumen montado · sin reconstruir imagen en cada cambio

En desarrollo, queremos que los cambios en el código se reflejen inmediatamente sin tener que reconstruir la imagen Docker. docker-compose.yml monta el directorio del proyecto como volumen y activa el modo --reload de uvicorn.

docker-compose.yml
services:
  api:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - .:/app           # monta el proyecto para hot-reload
    command: uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
    environment:
      - PYTHONUNBUFFERED=1

💡 Diferencia entre desarrollo y producción:
docker-compose.yml (desarrollo) sobrescribe el CMD del Dockerfile para agregar --reload y monta el código como volumen. En producción (Kubernetes), se usa la imagen construida con el CMD del Dockerfile — sin --reload y con el código embebido en la imagen.

    Base.metadata.create_all(bind=engine)

    if db.query(TaskDB).count() > 0:
