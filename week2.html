🏠 Cuenta GitHub (repositorio del código)

☁️ NUBE — github.com
GitHub aloja el repositorio que Jenkins monitorea cada 5 minutos (SCM polling). Cada git push que hagas desencadena automáticamente el pipeline.
  1. Ve a github.com/signup → crea tu cuenta gratuita.
  2. Crea un repositorio nuevo: New repository → nombre devops-taskapi → Public → sin README → Create repository.
  3. En tu terminal local, conecta el proyecto al repositorio:
    Terminal LOCAL
    # Desde la raíz del proyecto (ya tienes los commits de Semana 1)
git remote add origin https://github.com/TU-USUARIO/devops-taskapi.git
git push -u origin main
  4. Verifica en github.com/TU-USUARIO/devops-taskapi que los archivos aparecen.

📦 Cuenta Docker Hub (registro de imágenes)

☁️ NUBE — hub.docker.com
Docker Hub almacena la imagen Docker que Jenkins construye. Kubernetes la descarga desde aquí en Semana 3 con imagePullPolicy: Always.
  1. Ve a hub.docker.com/signup → crea tu cuenta gratuita.
  2. Elige un Docker ID (nombre de usuario). Este nombre aparecerá en todas tus imágenes: TU-USUARIO/devops-taskapi.
  3. Genera un Personal Access Token (PAT) — más seguro que tu contraseña:
    Account Settings → Security → Personal access tokens → Generate new token
    • Description: jenkins-pipeline
    • Permissions: Read & Write
    • Copia el token inmediatamente — solo se muestra una vez.
  4. En el Jenkinsfile, actualiza la variable con tu Docker ID: 
    environment {
    IMAGE_NAME = "TU-USUARIO/devops-taskapi"
}
  5. Confirma que puedes hacer login desde tu terminal:
    Terminal LOCAL
    docker login -u TU-USUARIO

jenkins/ — Imagen personalizada + docker-compose

Jenkins LTS con Docker CLI y Python 3 preinstalados

La imagen oficial jenkins/jenkins:lts-jdk17 no incluye ni el cliente de Docker ni Python 3. El pipeline necesita ambos para construir imágenes y ejecutar pytest. La solución es extender la imagen con un Dockerfile propio dentro de la carpeta jenkins/.

💡 ¿Por qué una carpeta jenkins/?
Separar los archivos de infraestructura CI del código de la aplicación mantiene el repositorio ordenado. El docker-compose.yml de la raíz levanta la aplicación; el de jenkins/ levanta el servidor CI.

jenkins/Dockerfile

jenkins/Dockerfile 
FROM jenkins/jenkins:lts-jdk17

USER root

# ── Docker CLI ──────────────────────────────────────────────
RUN apt-get update && \
    apt-get install -y ca-certificates curl gnupg lsb-release && \
    mkdir -p /etc/apt/keyrings && \
    curl -fsSL https://download.docker.com/linux/debian/gpg | \
        gpg --dearmor -o /etc/apt/keyrings/docker.gpg && \
    echo "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
         https://download.docker.com/linux/debian \
         $(lsb_release -cs) stable" > /etc/apt/sources.list.d/docker.list && \
    apt-get update && \
    apt-get install -y docker-ce-cli && \
    rm -rf /var/lib/apt/lists/*

# ── Python 3 + venv (para ejecutar pytest en el pipeline) ───
RUN apt-get update && \
    apt-get install -y python3 python3-pip python3-venv && \
    rm -rf /var/lib/apt/lists/*

# ── Acceso al socket de Docker ───────────────────────────────
RUN groupadd -f docker && usermod -aG docker jenkins

USER jenkins

jenkins/docker-compose.yml

jenkins/docker-compose.yml 
services:
  jenkins:
    build: .                         # construye la imagen desde ./Dockerfile
    container_name: jenkins
    restart: unless-stopped
    ports:
      - "8080:8080"               # interfaz web
      - "50000:50000"             # agentes JNLP
    volumes:
      - jenkins_home:/var/jenkins_home  # datos persistentes
      - /var/run/docker.sock:/var/run/docker.sock  # socket del host

volumes:
  jenkins_home:
⚠️ Socket de Docker
Montar /var/run/docker.sock permite que Jenkins ejecute comandos docker build y docker push usando el daemon del host. En producción considera usar Docker-in-Docker (dind) o Kaniko para mayor aislamiento.

Levantar Jenkins

terminal 
# Desde la raíz del repositorio:
cd jenkins
docker compose up -d --build

# Obtener la contraseña inicial de administrador:
docker exec jenkins cat /var/jenkins_home/secrets/initialAdminPassword

Configuración inicial de Jenkins

Setup Wizard, plugins y usuario administrador

Al acceder por primera vez a Jenkins, el asistente de instalación guía la configuración básica en cuatro pasos sencillos.

  1. Abre http://localhost:8080 y pega la contraseña del paso anterior en el campo Administrator password.
  2. Elige Install suggested plugins y espera a que finalice la instalación (2–5 minutos según conexión).
  3. Completa el formulario de creación del usuario administrador (username, password, email).
  4. Acepta la URL de Jenkins (http://localhost:8080/) y haz clic en Start using Jenkins.

Instalar plugins adicionales requeridos

Los plugins sugeridos no incluyen Docker Pipeline. Instálalo manualmente:

  1. Menú lateral → Manage Jenkins → Plugins → Available plugins.
  2. Busca Docker Pipeline y marca la casilla.
  3. Busca JUnit (si no se instaló con los sugeridos) y marca la casilla.
  4. Haz clic en Install y reinicia Jenkins cuando lo solicite.
🔎 Plugins mínimos para este pipeline
  • Git — clonar el repositorio (incluido en suggested)
  • Pipeline — soporte Jenkinsfile (incluido en suggested)
  • Docker Pipeline — pasos docker.build y withDockerRegistry
  • JUnit — archivar el reporte XML de pytest
  • Credentials Binding — paso withCredentials (incluido en suggested)

Credenciales Docker Hub en Jenkins

ID dockerhub-creds — Personal Access Token recomendado

El stage Push del pipeline necesita autenticarse con Docker Hub. Guardar las credenciales en Jenkins (nunca en el código) es la práctica estándar de CI/CD.

  1. Menú lateral → Manage Jenkins → Credentials → (global) → Add Credentials.
  2. Tipo: Username with password.
  3. Username: tu usuario de Docker Hub.
    Password: tu contraseña o (recomendado) un Personal Access Token con permiso Read/Write.
  4. ID: escribe exactamente dockerhub-creds (este valor se referencia en el Jenkinsfile).
  5. Haz clic en Create.
🔓 Personal Access Token (PAT)
En Docker Hub: Account Settings → Security → Personal access tokens → Generate new token. El PAT tiene alcance limitado y puede revocarse sin cambiar tu contraseña principal. Es la forma recomendada de autenticar pipelines CI/CD.

ID de la credencial: debe ser exactamente dockerhub-creds — cualquier otro valor hará fallar el stage Push porque el Jenkinsfile referencia ese ID.

Jenkinsfile — Estructura del pipeline + Stage Checkout

Pipeline declarativo con environment, triggers y SCM checkout

El archivo Jenkinsfile en la raíz del repositorio describe el pipeline completo usando la sintaxis declarativa de Jenkins. A diferencia de la sintaxis scripted, la declarativa impone una estructura clara y validación en tiempo de carga.

Repositorio Git
→
Jenkinsfile
→
Jenkins lee el pipeline
→
Ejecuta los 4 stages
Jenkinsfile — bloque raíz + Checkout 
pipeline {
    agent any          // ejecutar en cualquier nodo disponible

    environment {
        IMAGE_NAME = "TU-USUARIO/devops-taskapi"  // ← cambia tu-usuario
    }

    triggers {
        pollSCM('H/5 * * * *')   // SCM polling cada ~5 min
    }

    stages {

        stage('Checkout') {
            steps {
                checkout scm    // clona la rama configurada en el job
            }
        }

        // ... stages siguientes (Test, Build, Push)
    }

    // ... bloque post
}
💡 H/5 * * * * — SCM Polling con hash
El símbolo H distribuye los jobs uniformemente dentro del intervalo. Así, si hay 20 pipelines configurados con H/5, Jenkins no los ejecuta todos al mismo minuto, evitando picos de carga. Es una best practice frente a usar */5 * * * * exacto.

Stage Test — pytest + reporte JUnit XML

Entorno virtual, pip install silencioso y archivo test-report.xml

El stage Test crea un entorno virtual de Python dentro del workspace de Jenkins, instala las dependencias y ejecuta pytest con salida JUnit XML. El paso junit del bloque post publica el informe en la UI de Jenkins.

Jenkinsfile — stage Test 
        stage('Test') {
            steps {
                sh '''
                    python3 -m venv .venv
                    . .venv/bin/activate
                    pip install --quiet -r requirements.txt
                    pytest tests/ -v --junit-xml=test-report.xml
                '''
            }
            post {
                always {
                    junit 'test-report.xml'   // archivar en Jenkins UI
                }
            }
        }
💡 --junit-xml y el paso junit
  • pytest --junit-xml=test-report.xml genera un XML compatible con el estándar JUnit (origen: Java, adoptado por la mayoría de los CI servers).
  • El paso junit 'test-report.xml' lee el archivo y publica los resultados en la página del build: gráficas de tendencia, detalle de fallos, etc.
  • Está dentro de post { always { ... } } para que el informe se publique aunque el stage falle.

Stage Build — docker build + tag :latest

Imagen versionada con BUILD_NUMBER y alias latest

El stage Build construye la imagen Docker desde el Dockerfile de la raíz y genera dos etiquetas: una versionada con BUILD_NUMBER y el alias latest.

Jenkinsfile — stage Build 
        stage('Build') {
            steps {
                sh "docker build -t ${IMAGE_NAME}:${BUILD_NUMBER} ."
                sh "docker tag  ${IMAGE_NAME}:${BUILD_NUMBER} ${IMAGE_NAME}:latest"
            }
        }
💡 BUILD_NUMBER
Variable de entorno propia de Jenkins. Se incrementa automáticamente con cada ejecución del pipeline (1, 2, 3...). Usar este número como etiqueta de imagen permite tener un historial trazable: cada imagen en Docker Hub corresponde exactamente a un build de Jenkins.

Ejemplo: build #5 produce TU-USUARIO/devops-taskapi:5 y TU-USUARIO/devops-taskapi:latest.

Stage Push — withCredentials + docker login seguro

Credenciales inyectadas en memoria, password por stdin

El stage Push publica ambas etiquetas en Docker Hub. Las credenciales nunca aparecen en los logs gracias a withCredentials, y la contraseña se pasa por stdin para evitar que quede en la lista de procesos del sistema.

Jenkinsfile — stage Push 
        stage('Push') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'dockerhub-creds',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )]) {
                    sh '''
                        echo "$DOCKER_PASS" | docker login -u "$DOCKER_USER" --password-stdin
                        docker push ${IMAGE_NAME}:${BUILD_NUMBER}
                        docker push ${IMAGE_NAME}:latest
                    '''
                }
            }
        }

Bloque post del pipeline completo

Jenkinsfile — bloque post 
    post {
        always {
            sh 'docker logout || true'   // cerrar sesión siempre
            cleanWs()                    // limpiar workspace
        }
        success {
            echo "Pipeline OK — imagen ${IMAGE_NAME}:${BUILD_NUMBER} publicada"
        }
        failure {
            echo "Pipeline FAILED en build #${BUILD_NUMBER} — revisar logs"
        }
    }
💡 --password-stdin vs -p $PASS
Pasar la contraseña como argumento -p la expone en el output de ps aux (lista de procesos) mientras el proceso está activo. Con --password-stdin, la contraseña viaja por la entrada estándar y nunca aparece en el historial ni en los logs de Jenkins.
💡 cleanWs()
Borra el directorio del workspace de Jenkins al finalizar cada build. Evita acumulación de archivos entre builds (artefactos, cachés, DBs de test) y reduce el uso de disco en el servidor CI.

Configurar job en Jenkins + SCM Polling

New Item → Pipeline from SCM → GitHub URL + pollSCM

Con el Jenkinsfile commiteado al repositorio, crea el job en Jenkins para que lo lea automáticamente en cada cambio.

  1. Menú lateral → New Item. Escribe un nombre (p. ej. devops-taskapi), elige Pipeline y haz clic en OK.
  2. En la sección Build Triggers, marca Poll SCM y escribe el schedule: 
    H/5 * * * *
  3. En la sección Pipeline, selecciona Pipeline script from SCM.
    • SCM: Git
    • Repository URL: la URL de tu repositorio en GitHub
    • Branch: */main
    • Script Path: Jenkinsfile
  4. Haz clic en Save.
  5. Ejecuta el primer build manualmente: Build Now.
⚠️ Actualiza IMAGE_NAME en el Jenkinsfile
Antes de hacer push, edita la variable de entorno en Jenkinsfile: 
environment {
    IMAGE_NAME = "TU-USUARIO/devops-taskapi"
}
Reemplaza TU-USUARIO por tu Docker ID en Docker Hub.

📄 Jenkinsfile completo

Referencia rápida del archivo completo tal como queda en el repositorio:

Jenkinsfile 
pipeline {
    agent any

    environment {
        IMAGE_NAME = "TU-USUARIO/devops-taskapi"
    }

    triggers {
        pollSCM('H/5 * * * *')
    }

    stages {

        stage('Checkout') {
            steps {
                checkout scm
            }
        }

        stage('Test') {
            steps {
                sh '''
                    python3 -m venv .venv
                    . .venv/bin/activate
                    pip install --quiet -r requirements.txt
                    pytest tests/ -v --junit-xml=test-report.xml
                '''
            }
            post {
                always {
                    junit 'test-report.xml'
                }
            }
        }

        stage('Build') {
            steps {
                sh "docker build -t ${IMAGE_NAME}:${BUILD_NUMBER} ."
                sh "docker tag  ${IMAGE_NAME}:${BUILD_NUMBER} ${IMAGE_NAME}:latest"
            }
        }

        stage('Push') {
            steps {
                withCredentials([usernamePassword(
                    credentialsId: 'dockerhub-creds',
                    usernameVariable: 'DOCKER_USER',
                    passwordVariable: 'DOCKER_PASS'
                )]) {
                    sh '''
                        echo "$DOCKER_PASS" | docker login -u "$DOCKER_USER" --password-stdin
                        docker push ${IMAGE_NAME}:${BUILD_NUMBER}
                        docker push ${IMAGE_NAME}:latest
                    '''
                }
            }
        }
    }

    post {
        always {
            sh 'docker logout || true'
            cleanWs()
        }
        success {
            echo "Pipeline OK — imagen ${IMAGE_NAME}:${BUILD_NUMBER} publicada en Docker Hub"
        }
        failure {
            echo "Pipeline FAILED en build #${BUILD_NUMBER} — revisar logs"
        }
    }
}
